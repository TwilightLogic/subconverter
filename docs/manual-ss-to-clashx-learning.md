# 手动把 `ss://` 节点转成 ClashX / Clash Meta 配置

这份文档记录了我们在 fork 当前仓库后，通过一次真实排障学到的内容：如何把单条 `ss://` 分享链接手动转成 ClashX / Clash Meta 可导入的 YAML 配置，以及为什么这个方法能工作。

这不是 `subconverter` 的自动转换说明，而是一份学习文档。它适合用来理解节点格式、验证单个节点是否可用，以及帮助后续把人工流程对照到项目源码中的自动流程。

在真正开始前，先把一个看起来很尖锐、但其实很重要的问题说清楚：

> 既然 `subconverter` 已经能自动转换，为什么还要手动转换？

短答案是：手动路径不是为了替代自动转换，而是为了排障和建立心智模型。

当时混在一起的不确定项有很多：

- 节点本身到底是不是有效
- `ss://` 到底是单节点分享链接，还是订阅
- ClashX / Clash Meta 到底要吃什么格式
- `subconverter` 是不会转，还是我们不会调用
- 配置有没有被客户端真正加载
- 查 IP 的流量到底有没有真的经过代理

手动拆一遍的价值，在于把这条复杂链路拆开：

```txt
ss:// 节点
  ↓
解析节点字段
  ↓
构造最小 Clash 配置
  ↓
验证客户端是否真正走代理
```

这样一来，我们就能知道问题到底是在节点、转换、导入，还是代理激活这几层里的哪一层。

## 1. 背景：`ss://` 链接是什么

单条 Shadowsocks 节点通常长这样：

```txt
ss://BASE64(method:password@server:port)#name
```

或者：

```txt
ss://BASE64(method:password)@server:port#name
```

它表示的是一个节点分享链接，不是 Clash 订阅文档，也不是一份完整的 YAML 配置。

这条链接里真正有用的信息通常只有几类：

- `method` / `cipher`
- `password`
- `server`
- `port`
- `name`

我们的手动转换，本质上就是把这些字段提取出来，再按 Clash 的字段名重新组织。

## 2. 为什么不能直接把 `ss://` 当 Clash 订阅导入

这是这次排障里最关键的认知。

`ss://` 链接是“单节点分享格式”，而 ClashX / Clash Meta 更常吃的是：

- 一份完整的 Clash YAML
- 或者一个返回 Clash YAML 的订阅地址

所以：

```txt
ss:// 单节点分享链接
  !=
Clash 订阅文档
```

如果客户端没有直接支持把 `ss://` 作为单节点导入，就需要先把它翻译成 Clash YAML。

这里的思考重点不是“手动比自动更好”，而是：

```txt
如果自动链路不通，
先确认最小可用路径是什么
```

只要最小路径能跑通，我们就知道这个项目不是完全不工作，而是某个中间环节出了问题。

## 3. 手动转换流程

整体思路可以看成下面这条链：

```txt
ss:// 链接
  ↓
手动解析 Base64 和结构
  ↓
得到 server / port / cipher / password / name
  ↓
写成 Clash YAML 的 proxies
  ↓
把节点挂到 proxy-groups
  ↓
给出最小 rules
  ↓
导入 ClashX / Clash Meta
```

这条链其实就是我们把一个“黑盒问题”拆成多个白盒问题的过程：

- `ss://` 本身是不是合法
- 里面到底有哪些字段
- Clash 最小配置长什么样
- 客户端加载成功和真正代理生效是不是同一回事

如果这些问题都分开确认过，后面再回头用自动转换，心里就不会再是盲用状态。

### 3.1 从链接里取出字段

以下用脱敏后的示例说明：

```txt
ss://BASE64(aes-256-gcm:your-password@example.com:7793)#example-ss-node
```

人工解析后，最终会得到类似这样的字段：

- `name`: `example-ss-node`
- `server`: `example.com`
- `port`: `7793`
- `cipher`: `aes-256-gcm`
- `password`: `your-password`

这一小步的价值，主要不在“人肉解码”本身，而在于确认：

```txt
这个节点的最小必要信息到底是什么
```

一旦这几个字段都能稳定落出来，后面无论是手写 YAML，还是让 `subconverter` 自动转换，本质上都只是把这些字段映射到别的格式里。

### 3.2 写成最小可用的 Clash Meta 配置

下面这份配置适合先做“节点是否真正可用”的验证：

```yaml
mode: global
log-level: warning
ipv6: false

proxies:
  - name: example-ss-node
    type: ss
    server: example.com
    port: 7793
    cipher: aes-256-gcm
    password: your-password
    udp: true

proxy-groups:
  - name: GLOBAL
    type: select
    proxies:
      - example-ss-node
      - DIRECT

proxy-providers:

rule-providers:

rules:
  - MATCH,GLOBAL
```

这份配置里最重要的不是语法，而是结构：

- `proxies`: 定义节点本身
- `proxy-groups`: 定义客户端里真正可选择的组
- `rules`: 决定流量走向

如果只是写了 `proxies`，但没有把节点放进 `proxy-groups`，客户端里往往不会出现可切换的选项。

这一步对应的思考是：

```txt
我们不是在追求“最终最优配置”
而是在追求“最小可验证配置”
```

也就是说，这里故意用一个很简单的 `global` 方案，不是因为它适合长期使用，而是因为它最适合回答：

- 节点能不能被加载
- 节点能不能被选中
- 代理流量能不能真的出得去

## 4. 为什么 `proxy-groups` 很重要

这是这次排障里另一个非常具体的收获。

很多时候问题不是“节点不通”，而是：

```txt
节点定义成功
  !=
客户端里一定能选到它
```

只有当节点被放进一个组里，比如：

```yaml
proxy-groups:
  - name: GLOBAL
    type: select
    proxies:
      - example-ss-node
      - DIRECT
```

ClashX / Clash Meta 才会把它暴露成可切换的代理入口。

可以把它理解成：

```txt
proxies      = 原材料
proxy-groups = 用户真正操作的入口
```

这是我们这次排障里一个很关键的“认知翻面”。

一开始如果只盯着节点本身，很容易以为“节点写进配置就应该能用”。但实际用 ClashX / Clash Meta 时，客户端真正暴露给用户操作的，是组，不是裸节点。

所以这里解决的不是“如何再多写一段 YAML”，而是：

```txt
为什么节点明明存在，
客户端里却没有可切换的入口
```

## 5. 如何确认代理真的生效了

能看到节点，不代表流量一定经过它。

实际验证时，至少要确认下面几件事：

1. 这份 YAML 已经被 ClashX / Clash Meta 正确加载
2. `GLOBAL` 或对应组里，当前选中的确实是目标节点，不是 `DIRECT`
3. 路由模式适合测试。初次验证最简单的是 `mode: global`
4. 客户端已经开启系统代理，例如 `Set as System Proxy`

可以把这条链记成：

```txt
配置能加载
  ↓
节点能显示
  ↓
组里能选中
  ↓
系统代理已开启
  ↓
浏览器 / App 流量才会经过 Clash
  ↓
公网 IP 才会变化
```

这里最容易踩的坑，就是把“配置加载成功”和“流量真的经过代理”混为一谈。

这也是为什么我们后来会意识到，测速成功并不自动等于所有链路都已经正确：

- 可能节点可连，但没被当前组选中
- 可能组选中了，但系统代理没开启
- 可能系统代理开启了，但测试方式没有真正经过它

所以这一步的思考本质是：

```txt
把“看起来能用”和“真的在用”分开验证
```

## 6. 常见问题排查

### 6.1 `GLOBAL` 组里没有我的节点

优先检查：

- YAML 里 `proxy-groups` 是否为空
- 组里的 `proxies` 列表里是否真的写了节点名
- 节点名是否和 `proxies` 里的 `name` 完全一致
- 文件里是否混进了多余文本，导致 YAML 解析异常

### 6.2 测速成功，但查 IP 还是中国

这通常说明“节点能连”，但不一定说明“当前测试流量真的走了代理”。

优先检查：

- 当前组是否选中了节点而不是 `DIRECT`
- 当前是否处于 `global` 或符合预期的规则模式
- 是否打开了系统代理
- 查 IP 的浏览器或 App 是否真的使用系统代理

需要特别记住：

```txt
测速成功
  !=
所有流量都已走代理
```

这条经验直接改变了我们排查问题的顺序。

如果先相信测速页面，再去怀疑节点本身，排障会越走越乱。更有效的做法是把验证拆开：

1. 节点参数是不是正确
2. 配置是不是被客户端吃进去了
3. 当前组是不是选中了这个节点
4. 流量是不是经过系统代理

这样每一步都在回答不同的问题，而不是一次性押宝。

### 6.3 只有一条 `ss://`，这算配置了机场吗

更准确地说，这是“成功配置并使用了一条机场节点”，但还不是完整的订阅式机场配置。

区别在于：

- 手动方式：适合单个节点验证和学习
- 订阅方式：适合自动更新、多节点维护、长期使用

所以，如果把手动方案当成长期运营方式，那确实很笨；但如果把它当成一次“拆系统、建心智模型、定位问题”的方法，它就很有价值。

## 7. 这种手动方法适合什么时候

这条路径很适合以下场景：

- 只有一条单独的 `ss://` 节点，想尽快验证它是否可用
- 想理解 `ss://` 和 Clash YAML 之间是怎么映射的
- 想在阅读 `subconverter` 源码前，先建立正确的格式心智模型
- 想确认问题出在节点本身，还是出在客户端配置

但它不适合直接替代完整订阅方案，因为它没有自动处理：

- 机场节点变更
- 多节点更新
- 自动测速与分组维护
- 长期订阅同步

换句话说：

```txt
手动路径负责理解和诊断
自动路径负责日常使用和交付
```

这两条路不是互相竞争，而是先后关系。我们先用手动方法确认底层机制，再回到自动化主路径，才不会把“工具没问题”和“自己不会用”混在一起。

## 8. 这和项目里的自动流程是什么关系

这次手动方法，并不是发明了一条新的转换路径，而是在人工复现项目内部已经存在的思路。

项目里的自动流程可以粗略理解为：

```txt
ss:// 链接
  ↓
解析成统一内部节点结构 Proxy
  ↓
再导出成目标格式
  - clash
  - surge
  - quanx
  - 其他
```

这也是为什么，走过一遍手动路径之后，再看源码会突然清晰很多。

我们会意识到项目真正的核心不是“支持很多链接前缀”，而是这件事：

```txt
各种输入格式
  ↓
统一内部 Proxy 模型
  ↓
各种输出格式
```

也就是说，我们手动做的，并不是项目之外的旁门左道，而是在人工复现项目内部自动流程的一段核心逻辑。

和这次学习直接相关的源码位置有：

- [subparser.cpp](/Users/zibin/Downloads/My/app/subconverter/src/parser/subparser.cpp): 解析 `ss://`、`ssr://`、`vmess://` 等输入格式
- [subexport.cpp](/Users/zibin/Downloads/My/app/subconverter/src/generator/config/subexport.cpp): 把内部 `Proxy` 结构导出成 Clash 等目标格式

对 Shadowsocks 来说，我们手动做的事近似于：

```txt
ss://
  ↓
手动得到 method / password / server / port / name
  ↓
手动写成 Clash YAML
```

而项目自动做的是：

```txt
ss://
  ↓
explodeSS()
  ↓
Proxy
  ↓
proxyToClash()
  ↓
Clash YAML
```

## 9. 这次真正跑通的订阅转化链路

在真正接上机场订阅之后，我们最终跑通的并不是“Docker 容器自己去拉远程订阅”这条路，而是下面这条：

```txt
机场订阅链接
  ↓
宿主机先拉取订阅内容
  ↓
把内容包装成 data: URL
  ↓
交给本地 subconverter
  ↓
输出 Clash / Clash Meta 配置
```

这条链路的价值在于，它把“远程订阅下载”和“格式转换”分开了：

- 订阅下载：由宿主机完成
- 格式转换：由 `subconverter` 完成

这样我们就不再依赖 Docker 容器自己解析订阅域名。

对应到心智模型上，可以理解成：

```txt
自动转换本身没有失败
失败的是容器内的网络环境
```

这也是为什么我们最后没有放弃 `subconverter`，而是换了一种把输入交给它的方式。

## 10. 为什么 Docker 直拉订阅会失败

这次自动转换一开始一直报：

```txt
No nodes were found!
```

如果只看表面，很容易误判成：

- 订阅链接没有节点
- 机场格式不兼容
- `subconverter` 不支持

但从日志里看，真正的根因是：

```txt
Could not resolve host: jmssub.net
```

也就是说，问题卡在了最前面的 DNS 解析：

```txt
Docker 容器
  ↓
无法解析订阅域名
  ↓
无法下载订阅内容
  ↓
后续当然也解析不出节点
```

这一步非常值得记下来，因为它告诉我们：

```txt
“No nodes were found”
不一定是订阅内容为空
也可能是更前面的下载步骤根本没成功
```

所以以后再看到类似现象，排查顺序应该优先看：

1. 订阅链接在宿主机上能不能打开
2. 容器里能不能解析订阅域名
3. 下载到了什么内容
4. 最后才是格式兼容问题

## 11. fork 里的兜底脚本

为了绕过 Docker 容器 DNS 问题，这个 fork 新增了一个宿主机辅助脚本：

- [host_fetch_sub.py](/Users/zibin/Downloads/My/app/subconverter/scripts/host_fetch_sub.py)

它做的事情是：

```txt
宿主机请求远程订阅
  ↓
拿到订阅原始内容
  ↓
编码成 data: URL
  ↓
交给本地 subconverter
```

常用方式有两种。

打印一个本地可导入链接：

```bash
python3 scripts/host_fetch_sub.py 'https://example.com/subscription'
```

直接生成一份 Clash 配置文件：

```bash
python3 scripts/host_fetch_sub.py \
  --fetch-output \
  --output profile.yaml \
  'https://example.com/subscription'
```

这里推荐优先使用第二种，因为：

- 不需要在客户端里粘一个很长的本地 URL
- 便于先验证生成结果
- 更容易和普通配置导入流程对接

需要注意，这个脚本生成的本地 URL、YAML 文件和原始订阅内容都可能包含真实节点信息，因此都应该视为敏感数据，不要写进文档、commit、issue 或聊天记录。

## 12. 最终推荐工作流

结合这次经验，我们现在更清楚应该怎么使用这套工具链：

```txt
优先路径
机场订阅
  ↓
subconverter 直接转换
  ↓
Clash Meta 使用

容器 DNS 异常时的兜底路径
机场订阅
  ↓
宿主机 host_fetch_sub.py
  ↓
subconverter
  ↓
Clash Meta 使用
```

如果只是日常使用，订阅仍然应该是主路径；手动单节点配置和宿主机兜底脚本，分别适合：

- 手动单节点：理解底层、验证节点、定位问题
- 宿主机兜底脚本：容器网络异常时保持自动转换可用

这样一来，我们今天真正建立起来的就不是单一技巧，而是一套分层策略：

```txt
手动路径     -> 理解和诊断
自动路径     -> 日常使用
兜底脚本路径 -> 环境异常时的替代方案
```
